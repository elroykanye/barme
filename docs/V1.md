# Road to v1

v1 is a promise, not a feature list. A 1.x barme says three things:

1. **The on-disk format and native API are frozen.** Anything written to a 1.x
   server still reads on every later 1.x. Format changes ride a version stamp and
   a migration, never a silent break.
2. **You can trust it with data you can't afford to lose.** Survives a hard kill,
   stores no secret in the clear, ships with no standing default credential.
3. **It's operable.** Someone can run it, watch it, back it up, and recover it.

Distribution is explicitly **not** part of v1. See the bottom.

Current baseline: v0.3.0. Streaming ingest/egress land, GC and lifecycle already
run on a schedule, per-pot policy and multi-key auth exist. What's left is
hardening, not features.

---

## 1. Durability (the real gate)

A store that advertises integrity has to survive `kill -9` and power loss with no
half-written or dangling object.

- [x] **fsync the parent directory after every atomic rename.** `write_atomic`
  now fsyncs the containing directory after `persist`, and the grandparent when
  the shard directory is new, so the rename itself survives power loss — not just
  the bytes. Covers every store (all writes go through `write_atomic`). No-op on
  Windows; Linux is the deploy target. *(0.4.0)*
- [x] **Write ordering holds: chunks durable before the manifest, manifest before
  the pointer.** Confirmed in `finalize_write` (chunks → manifest → pointer →
  reverse); with durable writes each step is on disk before the next references
  it. The crash harness exercises it. *(0.4.0)*
- [x] **Crash/kill harness.** `scripts/crash-test.sh` runs concurrent uploads,
  `kill -9`s the daemon mid-write, restarts on the same data dir, and asserts
  every *acknowledged* object downloads and re-hashes clean. Proven: 8 crash
  cycles, 47 objects, zero lost or corrupted. *(0.4.0)*
- [x] **Startup recovery sweep.** On open the store reaps temp files a crashed
  write stranded and logs the count; the shard walkers skip non-chunk files so a
  stray temp file can't trip GC. A clean shutdown recovers zero. *(0.4.0)*
- [ ] **Decide fsync cost vs. a batched/async fsync mode** and document the
  guarantee (per-write durable by default). Deferred: current default is
  per-write durable, which is the safe one; a throughput mode can come later.

## 2. Security

Remove the three things the README currently apologizes for.

- [ ] **Hash secret keys at rest.** The key store and auth path hold the secret in
  plaintext (`barme-auth` compares `r.secret_key == secret`; `key_store` persists
  it raw). Store a salted hash (argon2 or scrypt), verify against it. SigV4 needs
  the raw secret to recompute the HMAC, so this needs a deliberate design: either
  keep SigV4 keys in a separately-protected store, or scope plaintext-required keys
  and hash the Basic-auth ones. Resolve this explicitly, don't hand-wave it.
- [ ] **No standing default credential.** `barme/barme` works out of the box today.
  Generate a random secret on first boot and print it once, or refuse to serve
  writes until a credential is set. Keep first-run friction low but not "anyone on
  the network is owner."
- [ ] **Presign verification.** Confirm expiring share links actually verify and
  reject past expiry (was a deferred piece).
- [ ] **Bound and audit the CORS / bind surface** before freezing config.

## 3. Format and API freeze

This is what makes the version number mean something.

- [ ] **Stamp the on-disk layout with a format version** and write a migration
  hook, so 1.1 can evolve the manifest without orphaning 1.0 data.
- [ ] **Freeze the manifest schema.** Lock field names and semantics; anything
  still in flux gets settled now.
- [ ] **Freeze the native API surface.** Path shapes, status codes, error bodies.
  Mark anything experimental clearly or pull it.
- [ ] **Write a compatibility test** that reads a fixture created by an older build.

## 4. Correctness

- [x] **GC safety under load.** Fixed a real data-loss race: a sweep could
  condemn and erase a streaming upload's chunks before its pointer committed
  (the "don't reap young chunks" guard the module doc promised wasn't
  implemented). Uploads now pin chunks in-flight and GC treats pins as reachable;
  proven with `grace=0` GC hammering and crash-plus-GC harnesses. *(0.4.1)*
- [x] **Concurrent writes keep every version.** Fixed a lost-update race on the
  read-modify-write pointer file (24 racing writers kept 1 version, not 24) with
  a per-key commit lock; `delete` takes the same lock so a delete can't be lost to
  a racing put. *(0.4.1, 0.4.2)*
- [x] **Derived state self-heals from corruption.** A corrupt condemned-set file
  no longer wedges GC forever (heals to empty), and a corrupt pointer line no
  longer orphans a whole key (intact versions are still read). *(0.4.2)*
- [ ] **verify at scale** already streams; add it to the crash harness so recovery
  and verify agree.
- [ ] **Fuzz key/pot names and content-type routing** against the 255-byte
  filename bound.

## 5. Ops

- [ ] **`/metrics`** (Prometheus): object/chunk counts, bytes, GC stats, error
  rates.
- [ ] **Health/readiness endpoint** distinct from liveness.
- [ ] **Backup + restore story.** At minimum: document that the data dir is a
  coherent backup target once fsync lands, and prove a copy restores clean.
- [ ] **Structured request logs** finalized (TraceLayer fields).

## Sequencing

Do it in this order. The first block can force a format change, so it goes before
anything that builds on the format.

1. **Durability** — dir-fsync, write-ordering test, crash harness, startup sweep.
   Ships as its own release (0.4.0). This is the block most likely to reshape the
   on-disk format, so it comes first, while the format is still cheap to change.
2. **Security** — secret hashing (with the SigV4 decision), kill the default
   credential, presign verify. (0.5.0)
3. **Format + API freeze** — version stamp, migration hook, schema/endpoint lock,
   compat fixture test. This is the actual 1.0-rc gate.
4. **Ops + correctness polish** — metrics, health, backup doc, GC-under-load test.
5. **1.0.0** — cut it, and mean the three promises at the top.

The existing console + config roadmap (`glistening-doodling-hearth.md`) is
parallel usability work. Land it around this, but it does not gate v1 — a
trustworthy headless node is v1.

## Explicitly v2, not v1: distribution

Distribution is the v2 headline, deferred on purpose. Content-addressing makes it
cheap to add *later*, which is exactly why it shouldn't hold up v1:

- Chunks are immutable and hash-named, so replication is "copy the chunks the peer
  doesn't have" — already the sync protocol. No mutation conflicts, no rebalancing.
- The path is incremental: async replication to N peers → read failover →
  quorum / anti-entropy. Each step ships alone.
- Pulling consensus, membership, and split-brain into v1 would balloon the test
  surface and freeze an API we'd regret.

v1 = a single node you'd trust. v2 = it scales out.
